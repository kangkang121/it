                      一、处理器架构的特定部分

                      在取指令数据的时候，处理器的内存管理单元需要把虚拟地址转换成物理地址。如果没有虚拟页没有映射到物理页，或者没有访问权限，处理器将生成页错误异常。

                      虚拟页没有迎合到物理页，这种情况通常称为缺页异常，有一下几种情况。

                      1）访问用户栈的时候，超出了当前用户站的范围，需要扩大用户栈。

                      2）当进程申请虚拟内存区域的时候，通常没有分配物理页，进程第一次访问的时候触发页错误异常。

                      3）内存不足的时候，内核把进程的匿名页换出到交换区。

                      4）一个文件页被映射到进程的虚拟地址空间，内存不足的时候，内个回收这个文件页，在进程的页表中删除这个文件的映射。

                      5）程序错误，访问没有分配给进程的虚拟内存区域。

                      前面四种情况，如果页错误异常处理程序成功的把虚拟页映射到物理页，处理程序返回后，处理器重新执行触发异常的指令。

                      第五种情况，页错误影响处理程序将发送段违法（SIFSEGV）信号杀死进程。

                       

                      没有访问权限，有以下两种情况：

                      1）可能是软件有意造成的，典型的例子是写时复制（Copy on Write CoW）：进程分叉生成子进程的时候，为了避免复制物理页，子进程和父进程以只读方式共享所有私有的匿名页和文件页。当其中一个进程试图写只读页时，触发页错误异常，页错误异常处理程序分配新的物理页，把旧的物理页的数据复制到新的物理页，然后把虚拟页映射到新的物理页。

                      2）程序错误，例如试图写只读的代码段所在的物理页。

                      第一种情况，如果页错误异常处理程序成功的把虚拟页映射到物理页，处理程序返回后，处理器重新执行触发异常的指令。

                      第二种情况，页错误异常处理程序将会发送段违法信号（SIGSEGV）杀死进程。

                      不同的处理器架构实现的也错误异常不同，页错误异常处理程序的前面一部分是各种处理器架构自定义部分，后面从函数handle_mm_fault开始的部分是所有处理器架构共用的部分。

                      生成页错误异常

                      假定ARM64处理器带有并开启内存管理单元。注如果没有开启内存管理单元不会转换虚拟地址，物理地址就是虚拟地址。

                      ARM64处理器在取指令或数据的时候，需要把虚拟地址转换成物理地址，分两种情况：

                      1）如果虚拟地址的高16位不是全1或全0（使用49=8位虚拟地址），是非法地址，生成页错误异常。

                      2）如果虚拟地址的高16位是全1或全0，内存管理单元根据关键字{地址空间标识符， 虚拟地值}查找TLB。

                      通常使用寄存器TTBR0_EL1的高16位存放正在纸吸管的进程的地址空间标识符（TTBR是 Translation Table Base Register的缩写 表示转换表基准寄存器，EL1是Exception Level 1de 缩写，表示异常级别1）。

                      寄存器TTBR1_El1存放内核的页群居目录的物理地址，寄存器TTBR0_EL1存放进程的页全局目录的物理地址。

                      如果命中了TLB表项，从TLB表项读取访问权限，金叉访问权限，如果没有访问权限，生成页错误异常。

                      如果没有明总TLB表项，内存管理单元将会查询内存中的页表，称为转换表遍历（translation table walk）分两种情况：

                      1）如果虚拟地址的高16位全部是1，说明内核时虚拟地址，应该查询内核的页表，从寄存器TTBR1_EL1取内核的页全局目录物理地址。

                      2）如果虚拟地址的高16位全部是0，说明用户虚拟地址，应该查询进程的页表，从寄存器TTBR0_EL1取进程的也全局目录的物理地址。

                      内存管理单元访问内存中的页表，会根据表项类型进行处理：

                      1）如果是无效描述符，生成页错误异常。

                      2）如果是块描述符或页描述符，把表项复制到TLB。

                      3）如果是表描述符，从表项读取下一级页表的物理地址，继续访问下一级页表。

                      处理页错误异常

                      在ARM64架构的系统中，用户程序在异常级别0运行，内核在异常级别1运行，异常级别0是用户模式，异常级别1是内核模式。

                      ARM64架构的内核定义了一个异常向量表，起始地址是vectors（源文件arch/arm64/kernel/entry.S），没给异常向量的长度是1128字节，但是在Linux内核中每个异常向量只有一条指令：跳转到对应的处理程序。异常向量表的虚拟地址存放在异常级别1的向量记住地址寄存器（Vctor Base Address Register for Exception Level 1）中。

                      如下图所示： 处理器生成页错误异常，页错误异常。页错误异常属于同步异常，处理器立即处理，从向量基准地址寄存器得到异常向量表的虚拟地址，然后根据异常类型选择对应的异常向量。

                      1. 如果异常类别是64位用户程序在异常级别0生成的同步异常，异常向量的偏移及时0x200，这个异常向量跳转到函数el1_sync。

                      2. 如果异常雷诺新格式64位用户程序在异常级别0生成的同步异常，异常向量的偏移是0x400，这个异常向量跳转到函数el0_sync。

                      3. 如果异常类型是32位用户程序在异常级别0生成的同步异常，异常向量的偏移是0x600，这个异常向量跳转到函数el0_sync_compat。

                       

                      以函数el0_sync为例说明，函数el0_sync根据异常级别1的影响症状寄存器的异常类别字段处理。

                      1）如果异常类别0生成的数据终止（data abort），即在异常级别0访问数据时生成页错误异常，那么调用函数el0_da。

                      2）如果异常类别是异常级别0生成的指令中止（instruuction abort），即在异常级别0取指令时生成页错误异常，调用函数el0_ia。

                      写时复制

                      有两种情况会执行写时复制（Copy on Write， CoW）。

                      1）进程分叉生成子进程的时候，为了避免复制物理页，子进程和父进程以只读方式共享所有私有does匿名映页和文件页。当其中一个进程试图写只读页时，触发页错误异常，页错误异常初六程序分配新的物理页，把就得物理页的数据复制到新的物理页，然后把虚拟页映射到新的物理页。

                      2）进程创建私有文件映射，然后读访问，触发页错误异常，异常处理程序把文件读到页缓存，然后以只读模式把虚拟页映射到文件的也缓存中的物理页。接着执行写访问，触发页错误异常，异常处理程序执行写时复制，为文件的也缓存重点额物流业创建一个副本，把缓存页映射到副本。这个副本是进程的私有匿名页，和文件脱离关系，修改副本不会导致文件变化。

                      三、内核模式页错误异常

                      内核访问虚拟内存地址，正常情况下不会出现虚拟页没有映射到物理页的状况，内核使用线性映射区域的虚拟地址，在内存管理子系统初始化的时候就会把虚拟地址映射到物理地址，运行过程中科能使用vmalloc()函数从vmalloc区域分配虚拟内存区域，vmalloc()哈数会分配并且映射到物理页。如果出现虚拟页没有映射到物理页的情况，一定是程序错误，内核将会崩溃。

                      内核可能访问用户虚拟地址，进程通过修通调用进入内核模式，有些系统调用会传入用户空间的缓冲区，内核必须使用头文件"uacess.h"定义的专用函数访问用户空间缓冲区，这些专用函数在异常表中添加了可能触发异常的执行地址和异常修正程序的地址。如果访问用户的缓冲区时生成页错误异常，页错误异常处理程序发现用户虚拟地址没有被分配给进程，就在异常表中查找指令地址对应的异常修正程序，如果找到了，使用异常修正程序修正异常，避免内核崩溃。

                      概括起来有3种处理方式：

                      1）如果不允许内核执行用户空间的指令，那么进程在内核模式下视图执行用户空间的指令时，内核崩溃。

                      2）如果进程在内核模式下访问用户虚拟地址，先使用函数__do_page_fault处理，如果处理失败，最后一招是使用函数__do_kernel_fault处理。

                      3）其他情况使用函数__do_kernel_fault处理。

                      异常表

                      进程在内核模式下运行的时候，可能需要访问用户虚拟地址，应用程序通常是不可信任的，不能保证传入的用户虚拟地址是合法的，所以必须采取措施保护内核。当前采用的措施使用异常表，每条表项有两个字段：

                      异常表通常不是直接保存虚拟地址，而是保存相对地址。假设可能触发异常的指令的虚拟地址p1，异常修正程序的其实虚拟地址是p2，对应的异常表项的字段insn的虚拟地址是p3，字段fixup的虚拟地址是p4，那么异常表项的字段insn的值是（p1 ~ p3）字段fixup的值是（p2 ~ p4）。
