                  二、三次握手细节分析

                  服务端代码，对socket执行的bind方法可以绑定监听接口，然后执行listen方法后，就会进入监听LISTEN状态。内核会为每一个处于LISTEN状态的socket分配两个队列，分别叫做半连接队列和全连接队列。

                  半连接队列、全连接队列是什么

                  • 半连接队列（SYN队列）：服务端收到第一次握手后，会将sock加入到这个队列中，队列内的sock都会与SYN_RECV状态。

                  • 全连接队列（ACCEPT队列）：在服务端收到第三次握手后，全将半连接队列的sock取出，放到全连接队列中。队列里的sock都处于ESTABLISHED状态。这里面的连接，就等着服务端执行accept()后被取出了。

                  看到这里，在建立连接的过程中根本不需要accept()参与，执行accept()只是为了从全连接队列取出一条连接。

                  虽然都叫队列，但其实全连接队列（icsk_accept_queue）是个链表，而半连接队列（syn_table）是个hash表。

                  为什么半连接队列要设计成hash表

                  先对比下全连接队列，本质上是个链表，因为也是线性的，说它是个队列也没毛病，它里面放的都是已经建立成的连接，这些正在等待被取走，而服务端取走连接的过程中，并不关系具体是哪个连接，只要是个连接就行，所以可以直接从对头取就行了，这个过程算法的时间复杂度位O(1)。

                  而半连接队列不一样了，因为队列都是不完整的额连接，等待第三次握手的到来，那么现在有一个第三次握手来，则需要从队列把相应IP端口的连接诶取出，如果半连接队列还是个链表，那我们就需要一次遍历，才能到我们想要的那个链接，时间复杂度就是O(n)。

                  如果半连接队列设计成hash的形式查找半连接的算法时间复杂度就是O(1)。

                  因此出于效率考虑，全连接队列被设计成链表，而半连接队列设计位hash表。

                  全连接队列满了会咋样

                  如果队列满了，服务端还是会收到客户端的第三次握手ACK，默认当人会丢弃这个ACK。

                  但除了丢弃这个之外，还有一些附带的行为，还会收到tcp+bnort_on_overfliow参数的影响。

                  2
                  1
                  # cat /proc/sys/net/ipv4/tcp_abort_on_overflow
                  2
                  0

                  • tcp_abort_on_overflow设置为0，全连接队列满了之后会丢弃这个第三次握手ACK包，并且开启定时器，重传第二次我收到SYN+ACK，如果重传超过一定限制次数，还会把对应的半连接队列的连接给删除。

                  • tcp_abort_on_overflow设置为1，全连接队列满了之后，就直接发RST给客户端，效果上看就是连接断开了。

                  这个现象是不是很熟悉，服务端口监听时，客户端尝试去连接，客户端尝试去连接，服务端也会回一个RST。这连个情况一样，所以客户端这时候收到RST之后，其实无法区分到底是端口为监听还是全连接满了。

                  半连接队列满了会咋样

                  一般会丢弃，但是这个行为可以通过tcp_syncookies参数控制，但比起这个，更重要的是先了解下半连接队列为什么会被打满？

                  首先我们需要明白，一般情况下，半连接的生存时间其实很短，只要在第一次和第三次握手间，如果半连接队列满了，说明服务端在疯狂的收到第一次握手情况，那可能遇到了SYN Flood攻击了。

                  所谓SYN Flood攻击你可以简单理解为，攻击方模拟客户端疯狂发第一次握手情况过来，服务端也会回复第二次握手过去，客户端死活不发第三次握手过来，这样做，可以把服务端半连接队列给打满，从而导致正常连接诶不能正常进来。

                  那这种情况怎么处理？有没有一种方法可以绕过半连接队列？

                  有，上面提到了tcp_syncookies排上用场了。

                  2
                  1
                  # cat /proc/sys/net/ipv4/tcp_syncookies
                  2
                  1

                  当它被设置为1的时候，客户端发送第一次握手SYN时，服务端不会放入到半连接队列，而是直接生成一个cookies，这个cookies会跟着第二次握手，发回客户端，客户端在第三次握手，客户端在发第三次我收到额时候就会带上这个cookies，服务端验证到它就是当初出发那个，就会建立连接并放入到全连接队列中。可以看出整个过程不需要半连接队列的参与。

                  会有一个cookies队列吗？

                  生成cookies，保存在哪儿？是不是有一个队列保存cookies？

                  我们可以反过来想一下，如果cookies队列，那它会跟半连接一样，到偷来，还是会被SYN Flood攻击打满。

                  实际上cookies并不会一直专门的队列保存，它是通过通信双方deIP地址端口、时间戳、MSS等信息进行实时计算的，保存在TCP包头的seq里。

                  当服务端收到客户端发来的第三次握手包时，会通过seq还原处通信双方的IP地址端口、时间戳、MSS，验证通过则建立连接。

                  cookies方案为什么不直接取代半连接队列？

                  目前看下来，syn_cookies方案省下了半连接队列所需要的队列内存，还能解决SYNFlood攻击，那为什么不直接取半连接队列？

                  凡是皆有利弊，cookies方案虽然可以防止SYNFloow攻击，但是也会有一些问题。因为服务端并不会保存连接信息，所以如果传输过程中数据丢包了，也不会重复第二次握手的信息。

                  另外，编码解码cookies都是比较耗CPU的，利用这一点儿，如果此时攻击者造大量的第三次握手包（ACK包）同事带上各种瞎编的cookies信息，服务端收到的ACK包后一位是正经的cookies，就过去解码了，耗CPU，最后发现不是正经的数据包才会丢弃。

                  这种通过狗仔大量ACK包区消耗服务端资源的额攻击，交ACK攻击收到攻击的服务器可能因为CPU资源耗尽导致不能响应正经的请求。

                  没有Listen为什么还能建立连接

                  既然没有accept方法能建立连接，那是不是没有listen方法也可以建立连接呢 ？ 是的，也可以链各个客户端同时给对方发出请求建立连接（TCP同时打开），这两个情况的共同点就是没有服务端参与，也就是没有listen，就能建立连接。

                  我们知道listen方法时，会创建半连接对哦列和全连接队列。

                  三次握手的过程中会在链各个队列中暂存连接信息。

                  所以形成连接诶，前提是你有个地方存放着，方便我收到额时候根据IP端口等信息找到soket信息。

                  那么客户端会有半连接队列吗？

                  没有，因为客户端没有之心listen因为半连接队列和全连接队列是在执行listen方法时，内核自动创建的。

                  但内核还有个全局hash表，可以存放sock连接的信息，这个全局hash表还四份ehash、bhash和listen_hash等。

                  在TCP自连接的情况中，客户端在connect方法时，最后将自己的看来你姐信息存放刀片这个全局hash表中然后将信息发出去，消息在经过地址重新回到TCP传输层的时候，就会根据IP端口信息，再一次从这个群居Hash中取出信息。于是握手包一来一回，最后建立成功。

                  三、总结

                  • 每一个socket执行listen'时内核都会自动创建一个半连接队列和全连接队列。

                  • 第三次握手前，TCP连接会放在半连接队列中，直到第三次握手到来，才会被放到全连接队列中。

                  • accept方法，只是为了从全连接队列中拿出一条连接诶，本身跟三次握手几乎毫无关系。

                  • 处于效率考虑，虽然都叫队列，，但半连接队列其实被设计成hash表，而全连接队列本质是链表。

                  • 全连接队列满了，可能因为收到了SYNFlood共计，可以设置tcp_syncookies绕开半连接队列。

                  • 客户端没有半连接队列和全连接队列，但有一个全局hash，可以通过它实现自连接或TCP同时打开。
