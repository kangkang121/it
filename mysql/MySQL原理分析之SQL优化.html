                filesort排序:
                61
                  -> 单路排序: 一次性取出满足条件行的所有字段, 然后在sort buffer中进行排序;用trace工具可以看到sort_mode信息里显示<sort_key, additional_fields>或者<sort_key,packed_additional_fields>.
                62
                  -> 双路排序: 回表排序, 根据相应的条件取出相应的排序字段和数据的行ID, 然后在sort buffer中进行排序, 排序完后再次去主键索引取回其它需要的字段;用trace工具可以看到sort_mode信息里显示<sort_key, rowid>.
                63
                  -> 参数控制: MySQL通过比较系统变量max_length_for_sort_data(默认1024字节)的大小和需要查询的字段总大小来判断使用哪种排序模式.
                64
                  --> 如果max_length_for_sort_data比查询字段的总长度大, 那么使用单路排序模式.
                65
                  --> 如果max_length_for_sort_data比查询字段的总长度小, 那么使用双路排序模式.

                152
                置小点, 让优化器选择使用双路排序算法.可以在sort_buffer中一次排序更多的行, 只是需要再根据主键回到原表取数据.
                153
                  -> 2. MySQL排序内存有条件可以配置比较大,可以适当增大max_length_for_sort_data的值,让优化器
                154
                优先选择全字段排序(单路排序), 把需要的字段放到sort_buffer中, 这样排序后就会直接从内存里返回查询结果了.
                155
                  -> 3. 注意: 如果全部使用sort_buffer内存排序一般情况下效率会高于磁盘文件排序, 但不能因为这个就随便增大sort_buffer(默认1M), mysql很多参数设置都是做过优化的, 不要轻易调整.
                156
                  -> 4. order by语句使用索引最左前列;使用where子句与order by子句条件列组合满足索引最左前列.
                157

                158
                  -> 6. 如果order by的条件不在索引列上, 就会产生Using filesort.
                159
                  -> 7. 能用覆盖索引尽量用覆盖索引.
                160
                  -> 8. 、group by与order by很类似, 其实质是先排序后分组, 遵照索引创建顺序的最左前缀法则. 对于group
                161
                by的优化如果不需要排序的可以加上order by null禁止排序. 注意, where高于having, 能写在where中的限定条件就不要去having限定了.

                二、分页查询优化

                15
                1
                case: 很多时候我们业务系统实现分页功能可能会用如下sql实现select * from employees limit 10000,10;
                2
                性能分析: 表示从表employees中取出从10001行开始的10行记录. 看似只查询了10条记录, 实际这条SQL是先读取 10010条记录, 然后抛弃前10000条记录, 然后读到后面10条想要的数据. 因此要查询一张大表比较靠后的数据, 执行效率是非常低的.
                3
                

                4
                优化方式1: 
                5
                  -> 方式: 根据自增且连续的主键排序的分页查询.
                6
                  -> 优化前: select * from employees limit 90000,5;
                7
                  -> 优化后: select * from employees where id > 90000 limit 5;
                8
                  -> 缺点: 这种优化方式只适用于表中id是连续自增的一旦出现有数据删除那么就会导致结果不一致.
                9
                

                10
                优化方式2:
                11
                  -> 方式: 根据非主键字段排序的分页查询.
                12
                  -> 优化前: select * from employees ORDER BY name limit 90000,5;
                13
                  -> 优化后: select * from employees e inner join (select id from employees order by name limit 90000,5) ed on e.id = ed.id;
                14
                  -> 优点: 内查询是走主键索引的, inner join关联主键的效率非常高. 而原SQL使用的是filesort 排序, 而优化后的SQL使用的是索引排序.

                三、Join关联查询优化

                56
                1
                数据准备:
                2
                  -> 创建表:
                3

                4

                7

                8

                10
                  create table t2 like t1;
                11
                  -> 数据准备: 往t1表插入1万行记录, 往t2表插入100行记录.
                12
                

                13
                mysql的表关联常见算法: Nested-Loop Join算法;Block Nested-Loop Join算法.
                14
                

                15
                Nested-Loop Join算法: 
                16
                  -> 介绍: 一次一行循环地从第一张表(称为驱动表)中读取行, 在这行数据中取到关联字段, 根据关联字段在另一张表(被驱动表)里取出满足条件的行, 然后取出两张表的结果合集.
                17
                  -> case: explain select * from t1 inner join t2 on t1.a = t2.a;
                18
                  id|select_type|table|type  |possible_keys|key  |key_len|ref       |rows |Extra        
                19
                  1 |  SIMPLE   |t2   | ALL  |idx_a        |NULL |  NULL |NULL      |100  |Using where
                20
                  1 |  SIMPLE   |t1   | ref  |idx_a        |idx_a|   5   |test.t2.a |1    |NULL
                21
                  -> case分析: 
                22
                  --> 驱动表是t2, 被驱动表是t1. 先执行的就是驱动表(执行计划结果的id如果一样则按从上到下顺序执行sql);优化器一般会优先选择小表做驱动表. 所以使用inner join时, 排在前面的表并不一定就是驱动表.
                23
                  --> 使用了NLJ算法. 一般join语句中, 如果执行计划Extra中未出现 Using join buffer则表示使用的join算法是 NLJ.
                24
                  -> sql执行的流程:
                25
                  --> 1.从表t2中读取一行数据.
                26
                  --> 2.取出关联字段a, 到表t1中查找.
                27
                  --> 3.取出表t1中满足条件的行, 跟t2中获取到的结果合并, 作为结果返回给客户端.
                28
                  --> 4.重复上面3步.
                29
                  -> 结论: 整个过程会读取t2表的所有数据(扫描100行), 然后遍历这每行数据中字段a的值, 根据t2表中a的值索引扫描 t1表中的对应行(扫描100次t1表的索引, 1次扫描可以认为最终只扫描t1表一行完整数据, 也就是总共t1表也扫描了100行). 因此整个过程扫描了200行.
                30
                

                31
                 Block Nested-Loop Join算法:
                32
                  -> 介绍: 基于块的嵌套循环连接, 把驱动表的数据读入到join_buffer中, 然后扫描被驱动表, 把被驱动表每一行取出来跟join_buffer中的数据做对比.
                33
                  -> case: explain select * from t1 inner join t2 on t1.b = t2.b;
                34
                  id|select_type|table|type  |possible_keys|key  |key_len|ref       |rows |Extra        
                35
                  1 |  SIMPLE   |t2   | ALL  |idx_a        |NULL |  NULL |NULL      |100  |Using where
                36
                  1 |  SIMPLE   |t1   | ref  |idx_a        |idx_a|  NULL |test.t2.a |10377|Using join buffer
                37
                  -> case分析: Extra中的Using join buffer(Block Nested Loop)说明该关联查询使用的是BNL算法.
                38
                  -> sql执行的流程:
                39
                  --> 1.把t2的所有数据放入到join_buffer(内存)中.
                40
                  --> 2.把表t1中每一行取出来, 跟join_buffer中的数据做对比.
                41
                  --> 3.返回满足join条件的数据整个过程对表t1和t2都做了一次全表扫描, 因此扫描的总行数为10000(表t1的数据总量)+100(表t2的数据总量)=10100. 并且join_buffer里的数据是无序的, 因此对表t1中的每一行, 都要做100次判断, 所以内存中的判断次数是100*10000=100万次.
                42
                    
                43
                思考:
                44
                  -> 被驱动表的关联字段没索引为什么要选择使用BNL算法而不使用Nested-Loop Join呢?  
                45
                  --> 如果上面第二条sql使用Nested-Loop Join, 那么扫描行数为100 * 10000 = 100万次, 这个是磁盘扫描. 
                46
                      很显然, 用BNL磁盘扫描次数少很多, 相比于磁盘扫描, BNL的内存计算会快得多.
                47
                      因此MySQL对于被驱动表的关联字段没索引的关联查询, 一般都会使用BNL算法. 如果有索引一般选择NLJ算法, 有索引的情况下NLJ算法比BNL算法性能更高.
                48
                

                49
                优化: 
                50
                  -> 关联字段加索引, 让mysql做join操作时尽量选择NLJ算法.
                51
                  -> 小标驱动大表, 写多表连接sql时如果明确知道哪张表是小表可以用straight_join写法固定连接驱动方式, 省去mysql优化器自己判断的时间.
                52
                  -> straight_join解释: straight_join功能同join类似, 但能让左边的表来驱动右边的表, 能改表优化器对于联表查询的执行顺序.
                53
                     比如: select * from t2 straight_join t1 on t2.a = t1.a;代表制定mysql选着t2表作为驱动表.
                54
                     straight_join只适用于inner join, 并不适用于left join, right join(因为left join，right join已经代表指定了表的执行顺序).
                55
                  -> 尽可能让优化器去判断, 使用straight_join一定要慎重, 因为部分情况下人为指定的执行顺序并不一定会比优化引擎要靠谱.

                四、in和exists优化

                23
                1
                in优化:
                2
                  -> 原则: 小表驱动大表, 即小的数据集驱动大的数据集.
                3
                  -> in: 当B表的数据集小于A表的数据集时, in优于exists.
                4
                  -> select * from A where id in (select id from B)
                5
                  #等价于：
                6
                    for(select id from B){ //先执行
                7
                       select * from A where A.id = B.id //在执行
                8
                   }
                9
                  
                10
                exists优化:
                11
                  -> exists ：当A表的数据集小于B表的数据集时，exists优于in 
                12
                    将主查询A的数据，放到子查询B中做条件验证，根据验证结果（true或false）来决定主查询的数据是否保留
                13
                  -> select * from A where exists (select 1 from B where B.id = A.id)
                14
                  #等价于:
                15
                  for(select * from A){//先执行
                16
                    select * from B where B.id = A.id
                17
                  }
                18
                  -> 优化建议:
                19
                  --> #A表与B表的ID字段应建立索引
                20
                  --> 1.EXISTS(subquery)只返回TRUE或FALSE,因此子查询中的SELECT * 也可以用SELECT 1替换, 官方说法是实际执行时会忽略SELECT清单, 因此没有区别.
                21
                  --> 2.EXISTS子查询的实际执行过程可能经过了优化而不是我们理解上的逐条对比.
                22
                  --> 3.EXISTS子查询往往也可以用JOIN来代替, 何种最优需要具体问题具体分析.

                思考: 
                7
                  -> 下边查询语句哪个更高?
                8
                  --> select count(1) from employees;
                9
                  --> select count(id) from employees;
                10
                  --> select count(name) from employees;
                11
                  --> select count(*) from employees;
                12
                

                13
                结论:
                14
                  -> 在MySQL5.7之前count(1) > count(name) >= count(*) > count(id).
                15
                     但是在MySQL5.7以后版本四个sql的执行计划一样, 说明这四个sql执行效率应该差不多, 区别在于根据某个字段count不会统计字段为null值的数据行. 最终使用的都是辅助索引.
                16
                  -> 为什么mysql最终选择辅助索引而不是主键聚集索引?
                17
                  --> 因为二级索引相对主键索引存储数据更少, 检索性能应该更高.
                18
                

                19
                常见优化方法:
                20
                  -> 1.查询mysql自己维护的总行数.
                21
                     对于myisam存储引擎的表做不带where条件的count查询性能是很高的. 因为myisam存储引擎的表的总行数会被mysql存储在磁盘上, 查询不需要计算.
                22
                    因为MyISAM引擎不支持事务, 因此不会出现数据不一致情况, 索引MySIAM会在内存中维护一份总数量值.
                23
                  -> 2.show table status语句: 如果只需要知道表总行数可以使用 show table status like '表名'.
                24
                  -> 3.将总数维护到Redis里: 插入或删除表数据行的时候同时维护redis里的表总行数key的计数值(用incr或decr命令), 但是这种方式可能不准, 很难保证表操作和redis操作的事务一致性.
                25
                  -> 4.增加计数表: 插入或删除表数据行的时候同时维护计数表, 让他们在同一个事务里操作.
